# «Технический долг. Устранение долга»

## 1. Проинспектировать проект на наличие технического долга

В рамках проекта «Dance Center Service» (Сервис управления танцевальным центром) был проведен анализ технического состояния:

| Технический долг                                                 | В нашем проекте                                                                                                                                                                                          | 
|------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **непонятный / нечитабельный код**                               | В некоторых сервисах (например, `AuthServiceImpl.java`) присутствует сложная логика выдачи и валидации JWT без подробных комментариев. На фронтенде встречаются перегруженные компоненты.                  |
| **дублирующийся код**                                            | Логика REST-запросов на фронтенде к бэкенду размазана по компонентам (с помощью `fetch`), отсутствует единый клиент. Множество разрозненных `.css` файлов (например, `TrainersPage.css`).                |
| **отсутствие автоматизации (тестов, сборки, развёртывания)**     | Отсутствует настроенный CI/CD пайплайн. Развёртывание происходит локально/вручную через `docker-compose`. Полное отсутствие автоматических тестов для фронтенд-приложения (React).                       |
| **запутанная архитектура и ненужные сложные зависимости**        | Логика обработки и маршрутизации JWT на клиенте частично пересекается и иногда дублирует проверки на сервере; риск возникновения циклических зависимостей в Spring-компонентах.                          | 
| **медленные / неэффективные средства**                           | Существует вероятность проблемы N+1 запросов в Hibernate при загрузке расписания занятий вместе со связанными сущностями (тренеры, залы, группы).                                                        |  
| **незакоммиченый код / долгоживущие ветки**                      | Разработка ведётся в одной ветке (main), не используются механизмы pull/merge requests для код-ревью.                                                                                                    | 
| **отсутствие / несоответствие технической документации**         | Библиотека Swagger (`springdoc-openapi`) подключена, однако контроллерам не хватает подробных описаний (аннотации `@Operation`, `@ApiResponses`), из-за чего документация генерируется неполной.         | 
| **отсутствие тестовой среды**                                    | Вся разработка и тестирование проходят на локальных машинах (localhost), отдельный сервер для предрелизного тестирования (staging) отсутствует.                                                          | 
| **длинные циклы интеграции / отсутствие непрерывной интеграции** | Интеграция и проверка работоспособности (тесты) запускаются разработчиками вручную. Непрерывная поставка отсутствует.                                                                                    |

## 5. Сделать выводы и обосновать необходимость (отсутствие таковой, частичную необходимость) устранения долга, внести коррективы в план разработки проекта

**Обоснование необходимости устранения:**
- **Отсутствие CI/CD и тестов на фронтенде:** По мере усложнения UI (добавление новых ролей и дэшбордов) ручное тестирование будет занимать недопустимо много времени. Автоматизация критична для скорости поставок.
- **Оптимизация базы данных:** Без решения проблемы N+1 приложение будет испытывать серьезные проблемы с производительностью при росте базы клиентов и расширении расписания.
- **Единый API-клиент:** Поможет легко и надежно перехватывать логику обновления токенов, что устранит баги с аутентификацией пользователей.

**Риски при игнорировании долга:**
- Риск регрессии: При внедрении новых систем (например, онлайн-оплаты) существует высокий риск сломать текущий функционал без покрытия тестами.
- Замедление разработки: Разрозненный CSS и дублирующиеся API-вызовы будут требовать всё больше времени на внесение изменений.
- Деградация производительности.

**Вывод:** В проекте «Dance Center Service» накопился умеренный технический долг, который в основном сконцентрирован на фронтенд-архитектуре и процессах DevOps. Значение 0.71 указывает на то, что долг начинает становиться ощутимым. На данном этапе его нельзя полностью игнорировать. В план разработки следует закладывать около 20-30% времени спринта на постепенное закрытие долга (в первую очередь: создание единого API-интерфейса и настройка базового CI пайплайна), параллельно с реализацией новых бизнес-фич.
