# 1 Введение

<a name="appointment"/>

## 1.1 Назначение
В этом документе описаны функциональные и нефункциональные требования к приложению «Администрирование танцевальной студией» для ОС Windows. Этот документ предназначен для команды, которая будет реализовывать и проверять корректность работы приложения.

<a name="business_requirements"/>

## 1.2 Бизнес-требования

<a name="initial_data"/>

### 1.2.1 Исходные данные
Различные танцевальные студии, где есть множество преподавателей и групп, составляют расписание, ведут учет посещений, хранят список танцоров и тренеров в различных неспециализированных сервисах, не приспособленных для танцев. 

<a name="business_opportunities"/>

### 1.2.2 Возможности бизнеса
Внедрение системы открывает возможности для автоматизации рутинных операций: составления расписания без конфликтов, ведения базы клиентов и тренеров, а также анализа загрузки залов. Это напрямую ведет к сокращению административных затрат, повышению эффективности использования ресурсов и принятию управленческих решений на основе данных, что увеличивает рентабельность и облегчает масштабирование бизнеса.
<a name="project_boundary"/>

### 1.2.3 Границы проекта
Проект будет строго сфокусирован на базовых функциях администрирования: управлении сущностями (тренеры, залы, танцоры, группы) и построении расписания с проверкой на конфликты. На первом этапе исключаются модули онлайн-записи клиентов, биллинга, автоматических SMS-уведомлений и аналитических панелей, чтобы обеспечить быстрый запуск и стабильную работу ядра системы.
<a name="analogues"/>

## 1.3 Аналоги
Прямыми аналогами являются такие продукты, как Mindbody и ClubReady, которые предлагают комплексные решения для управления фитнес- и велнес-бизнесом, включая запись, платежи и маркетинг. Их ключевое отличие от нашего проекта — это сложность и высокая стоимость, тогда как наше решение фокусируется на более простом и доступном инструменте, идеально подходящем для малых и средних танцевальных студий, которым не нужен избыточный функционал.

# 2 Требования пользователя

<a name="software_interfaces"/>

## 2.1 Программные интерфейсы
На первом этапе интеграция с внешними системами не предусмотрена в целях соблюдения границ проекта. Однако архитектура будет предусматривать возможность будущего расширения. Будет заложена абстракция для модуля уведомлений (например, INotificationService), что в перспективе позволит подключить сервисы SMS (например, Twilio) или email (например, SendGrid). Аналогично, будет спроектирован интерфейс для потенциальной интеграции с платежными системами (например, Stripe, CloudPayments) для автоматизации приема оплаты за занятия.

### **Системные требования**

<a name="functional_requirements"/>

## 3.1 Функциональные требования

<a name="main_functions"/>

### 3.1.1 Основные функции

<a name="user_management"/>

#### 3.1.1.1 Управление сущностями
**Описание.** Авторизованный администратор имеет возможность управлять основными сущностями системы.

| Функция | Требования | 
|:---|:---|
| Управление танцорами | CRUD-операции для базы танцоров (ФИО, контакты, группа) |
| Управление тренерами | CRUD-операции для базы тренеров (ФИО, специализация, контакты) |
| Управление залами | CRUD-операции для базы залов (название, вместимость, оборудование) |
| Управление группами | Создание/редактирование групп с привязкой к тренеру и списку танцоров |

<a name="schedule_management"/>

#### 3.1.1.2 Управление расписанием
**Описание.** Система предоставляет инструменты для составления и редактирования расписания занятий.

| Функция | Требования | 
|:---|:---|
| Создание занятия | Привязка группы, тренера, зала и времени с проверкой на конфликты |
| Просмотр расписания | Отображение в виде недельной сетки с фильтрацией по залам/тренерам |
| Редактирование расписания | Возможность переноса, отмены занятий с уведомлением участников |

<a name="reports"/>

#### 3.1.1.3 Отчетность
**Описание.** Система формирует базовые отчеты по деятельности студии.

| Функция | Требования | 
|:---|:---|
| Загрузка танцоров | Отчет по количеству танцоров по группам |
| Загрузка залов | Отчет по использованию залов за выбранный период |

<a name="restrictions_and_exclusions"/>

### 3.1.2 Ограничения и исключения
1. Система не включает модуль онлайн-записи для клиентов;
2. Отсутствует интеграция с платежными системами;
3. Рассылка уведомлений осуществляется вручную через внешние сервисы.

<a name="non-functional_requirements"/>

## 3.2 Нефункциональные требования

<a name="quality_attributes"/>

### 3.2.1 Атрибуты качества

<a name="requirements_for_ease_of_use"/>

#### 3.2.1.1 Требования к удобству использования
1. Интерфейс интуитивно понятен администраторам без технической подготовки;
2. Время обучения работе с системой не превышает 2 часов;
3. Критические действия защищены подтверждением.

<a name="security_requirements"/>

#### 3.2.1.2 Требования к безопасности
Резервное копирование данных ежедневно.

<a name="performance_requirements"/>

#### 3.2.1.3 Требования к производительности
1. Отклик интерфейса на действия пользователя - не более 1 секунды;
2. Одновременная работа до 5 администраторов в системе;
3. Поддержка базы данных до 10 000 записей.

<a name="external_interfaces"/>

### 3.2.2 Внешние интерфейсы
1. Веб-интерфейс адаптирован для разрешений от 1280x720 пикселей;
2. Поддержка современных браузеров (Chrome 90+, Firefox 88+, Safari 14+);
3. REST API для потенциальной интеграции с мобильным приложением.

<a name="restrictions"/>

### 3.2.3 Ограничения
1. Система реализована как веб-приложение;
2. Основная СУБД - PostgreSQL 12+;
3. Бэкенд реализован на Java 17+, фронтенд - React 18+.
