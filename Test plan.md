## Введение

Данный документ описывает план тестирования приложения "Администрирование танцевальной студией". Документ предназначен для людей, выполняющих тестирование данного проекта. Цель тестирования - проверка соответствия реального поведения программы проекта и ее ожидаемого поведения, которое описано в [требованиях](https://github.com/bootgussy/zhcrpo/blob/main/docs/requirements/RussianRequirementsDocument.md).

<a name="items"/>

## Объект тестирования

В качестве объектов тестирования можно выделить следующие функциональные требования:

* Управление танцорами (CRUD-операции)
* Управление тренерами (CRUD-операции) 
* Управление залами (CRUD-операции)
* Управление группами (создание/редактирование с привязкой к тренеру и танцорам)
* Создание занятий в расписании с проверкой на конфликты
* Просмотр расписания в виде недельной сетки
* Редактирование расписания (перенос, отмена занятий)
* Формирование отчетов по загрузке танцоров и залов

<a name="quality"/>

## Атрибуты качества

1. Функциональность:
    - функциональная полнота: приложение должно выполнять все заявленные функции;
    - функциональная корректность: приложение должно выполнять все заявленные функции корректно;
2. Удобство использования:
    - интерфейс интуитивно понятен администраторам без технической подготовки;
    - время обучения работе с системой не превышает 2 часов;
    - критические действия защищены подтверждением.
3. Производительность:
    - отклик интерфейса на действия пользователя - не более 1 секунды;
    - поддержка одновременной работы до 5 администраторов;
    - поддержка базы данных до 10 000 записей.

<a name="risk"/>

## Риски

В данном случае к рискам можно отнести:
* невозможность работы системы при превышении лимита в 10 000 записей;
* снижение производительности при одновременной работе более 5 администраторов;
* конфликты расписания при некорректной проверке пересечений занятий.

<a name="features"/>

## Аспекты тестирования

В ходе тестирования планируется проверить реализацию основных функций приложения. Аспекты, подвергаемые тестированию:

### Управление танцорами
Этот вариант использования необходимо протестировать на:
* создание нового танцора;
* просмотр информации о танцоре;
* редактирование данных танцора;
* удаление танцора;
* проверку невозможности добавления уже существующего танцора.

### Управление тренерами
Этот вариант использования необходимо протестировать на:
* создание нового тренера;
* просмотр информации о тренере;
* редактирование данных тренера;
* удаление тренера.

### Управление залами
Этот вариант использования необходимо протестировать на:
* создание нового зала;
* просмотр информации о зале;
* редактирование данных зала;
* удаление зала.

### Управление группами
Этот вариант использования необходимо протестировать на:
* создание новой группы;
* привязку тренера к группе;
* добавление танцоров в группу;
* редактирование состава группы;
* удаление группы.

### Управление расписанием
Этот вариант использования необходимо протестировать на:
* создание занятия с привязкой группы, тренера, зала и времени;
* проверку на конфликты расписания (тренер, зал);
* просмотр расписания в виде недельной сетки;
* фильтрацию расписания по залам/тренерам;
* перенос занятия;
* отмену занятия;
* проверку невозможности добавления нескольких групп в одну ячейку расписания.

### Отчетность
Этот вариант использования необходимо протестировать на:
* формирование отчета по количеству танцоров по группам;
* формирование отчета по использованию залов за выбранный период;
* корректность данных в отчетах.

### Нефункциональные требования
* проверка отклика интерфейса (не более 1 секунды);
* проверка работы интерфейса в разрешениях от 1280x720 пикселей;
* проверка интуитивной понятности интерфейса;
* проверка подтверждения критических действий.

<a name="approach"/>

## Подходы к тестированию

При тестировании будет использован ручной подход. Тестирование будет включать:
* функциональное тестирование всех CRUD-операций;
* тестирование пользовательского интерфейса на соответствие мокапам;
* тестирование производительности;
* тестирование обработки граничных условий и исключительных ситуаций.

<a name="pass"/>

## Представление результатов

Результаты тестирования будут представлены в документе "Результаты тестирования", который будет включать:
* описание тестовых сценариев;
* ожидаемые и фактически полученные результаты;
* обнаруженные дефекты с указанием критичности;
* рекомендации по исправлению.

<a name="conclusion"/>

## Выводы

Данный тестовый план позволяет протестировать основной функционал приложения "Администрирование танцевальной студией". Успешное прохождение всех тестов не гарантирует полной работоспособности во всех сценариях использования, однако позволяет полагать, что данное программное обеспечение работает корректно в соответствии с заявленными требованиями. Тестирование должно быть проведено в соответствии с планом, а результаты должны быть проанализированы для принятия решения о готовности приложения к использованию.
